## 1. 项目目标与最终结果

<!-- 说明最初选择 Coding Agent Harness 方向的原因。简述最终完成了什么。不要写成 README，需要加入个人判断。 -->

做这个项目前，我先问自己：一个编码智能体最稀缺的能力是什么？不是"能写更多代码"，而是"在错误动作真正落地之前把它拦下来"。多数 Agent 演示把重心放在能力上限——让模型尽量自由地改文件、跑命令；我选择把重心放在行为下限——每个动作都必须先过治理闸门。于是 AegisCode 被定义为**治理优先**的 harness：命令词法治理、路径围栏、四档判定、人类在环审批、审计哈希链是最深的第一支柱；客观反馈闭环（pytest / 退出码 / 文件变更范围驱动自我修正）是完整但不追求最深的第二支柱。

最终结果：自实现的 `HarnessCore` 主循环（不依赖现成 Agent SDK）跑通，六个维度都有最低实现、治理维度深化；`make test`（444 passed）与 `make demo`（三项机制演示）全程 MockLLM 驱动、零网络；公网 demo URL 上线；真实 LLM Provider 也补齐到可实际驱动 Harness。

## 3. Harness Core 与治理机制的设计取舍

<!-- 说明为什么将治理作为重点维度。分析 Agent Loop、工具协议、审批、路径边界等设计选择。写出至少一个曾经考虑但最终放弃的方案。 -->

治理作为重点维度，是因为它是"harness 相对于裸调用 LLM"最有价值的部分。几个关键选择：主循环是显式 `while` 循环，每轮只允许一个结构化动作，便于逐动作治理与审计；完成判定交给 `final_verifier` 客观复跑 pytest，而非模型自称"做完了"；审批绑定到动作指纹，改动过的动作使旧批准失效（SUPERSEDED），杜绝"批一个、执行另一个"。审计链坚持 redact-before-hash——脱敏发生在计算 SHA256 与落库之前，哈希不会算在明文上。

放弃的方案举例：路径围栏最初考虑"绝对路径一律拒绝"，冷启动讨论（第 13 轮）后改为"绝对路径经 realpath 归属判定，落在工作区内才放行"，因为一刀切会误伤合理用法。

## 4. MockLLM、确定性测试与三个 Demo

<!-- 说明为什么课程强调移除真实 LLM 后仍能测试。分析 MockLLM 给开发和调试带来的价值。说明三个 Demo 分别验证了什么。 -->

课程强调"移除真实 LLM 后每个核心机制仍能确定性测试"，这个约束逼我把机制做成**代码**而不是"提示词祈祷"。MockLLM 按脚本回放响应、零网络零 Key，让治理、反馈、审批、停机、审计都能被确定性单测覆盖，也让调试可复现——不用为一个偶发的模型输出反复烧钱重跑。

三个 Demo 分别验证：Demo 1 危险命令在执行前被 DENY（工具执行次数为 0）；Demo 2 测试失败反馈驱动模型改变下一步动作，完成由客观复跑判定；Demo 3 审批绑定与失效（批准原动作后执行，改动参数后旧批准 SUPERSEDED）。

## 5. 真实 LLM Provider 追加实现

<!-- 明确这不是课程强制项。说明为什么 Adapter 存在仍不足以代表真实可用。描述系统提示词、工具协议和真实交互完善带来的认识。说明真实模型非确定性与 MockLLM 测试之间的关系。 -->

**这不是课程强制项**——课程验收依赖 MockLLM 确定性测试和三个机制 Demo，从未要求真实 LLM 端到端测试。初始版本已有可注入的 LLM 抽象、MockLLM 和真实 Provider Adapter 框架，但真实 CLI 一跑就发现：Adapter 存在 ≠ 真实可用。缺的是系统提示词（模型不知道自己是谁、能用什么工具、边界在哪）、从 Tool Registry 动态生成的工具协议、上下文构建和错误反馈引导。

补齐后，第一次真实运行（DeepSeek）又暴露一个真实现象：模型测试通过后不 `finish`、反复重写相同文件，被 NO_PROGRESS 停机——治理和停机机制都正确工作，问题是真实模型的非确定性。这让我更清楚：真实模型验证有价值，但**不能替代** MockLLM 测试；确定性测试才是回归可信的基础。

## 6. 部署、WebUI 与交付过程

<!-- 如实说明 Render 是原计划任务，但第一次没有完成。分析后续恢复部署、补充 Web Demo 和 UI 优化的过程。说明公网环境带来的安全边界问题。 -->

Render 公网部署是原计划要求，第一次没做完、暂缓，后续恢复完成（`render.yaml`、`/healthz`、Demo Mode、安全沙箱、公网 URL、`make deploy-check`）。之后为公网演示的完整性，补充了 WebUI 三项预设 MockLLM Demo（复用 `make demo` 同一套场景与成功条件），并做了一次黑白灰的纯表现层 UI 优化——没有改布局、Harness、API、治理或 Demo 逻辑。

公网环境带来独特的安全问题："谁能来我的服务器执行代码"。对策是演示沙箱 + MockLLM + 收紧的工作区与命令白名单，治理引擎在演示模式下也不关闭。

## 7. Superpowers、Subagent 与开发工作流

<!-- 说明 brainstorming、writing-plans、TDD、review、worktree 等工作流的实际作用。不要只罗列 Skill 名称。分析哪些环节真正提高了质量，哪些增加了协调成本。 -->

整个项目走子智能体驱动开发：每个里程碑一个 worktree、每个 task 一个全新子智能体、严格 TDD（先写失败测试→确认失败→最小实现→确认通过）、每个 task 后两阶段评审。真正提高质量的是 TDD + 评审闭环——它不止一次当场抓出"测试其实没测到真实路径"的问题：例如一个断言即使删掉被测规则也仍然通过的同义反复测试，是评审要求做 mutation 验证后才现形的。

增加协调成本的是子智能体在大文件、前端整合类任务上容易 context 溢出，这类任务改由控制器直接做、再派独立评审更稳。

## 8. 失败、人工干预与最重要的经验

<!-- 至少选择 2～3 个真实问题，说明原因、解决过程和后续如何避免。 -->

至少三个真实问题：

1. **计划中的部署任务未按时完成**：Render 是原计划要求却被暂缓，文档一度留下"待部署"。后续恢复完成，并在文档里如实标注"原计划→暂缓→恢复→完成"，而不是改写成"新增任务"。
2. **单测绿 ≠ 端到端接上**：记忆检索和审批指纹绑定都建好、都有单测，却一度没真正接进主循环（`harness` 里 `memories=[]` 硬编码）。只有把评审拉到"整条活循环端到端"的粒度才发现闭环缺了一半。相关的还有命令治理旁路（`python3` 绕过只匹配 `python` 的规则、`run_command` 未过路径围栏可读 `.env`），是拿真实默认配置逐条复现才确认的。
3. **真实 Provider 只有框架、无法实际交互**：Adapter 在但缺提示词与工具协议。定义为 enhancement 并补齐后解决；且用确定性单测守护，真实 E2E 只作人工触发。

已知限制（未来方向，非已完成）：锁定 Python + pytest 单栈；记忆是 SQLite 最小实现、无向量库；写快照回滚为 v2、当前 no-op；密钥扫描器仅覆盖已知格式；本地面板无鉴权（仅本机 / 公网仅 Demo 沙箱）。这些多数属于"可增强"而非"必须修复"。真正需要警惕的是任何 fail-open 的治理默认值——冷启动第四轮就挖出过一个符号链接旁路的敏感文件绕过。若继续做，我最想补的是把治理决策也纳入记忆（在"永不作治理依据"的前提下做可解释提示），以及给 WebUI 加一层真正的鉴权再谈更开放的公网使用。
