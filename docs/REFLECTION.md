# 项目反思 · AegisCode

> 本文是初稿,供后期个人润色。内容基于 `SPEC_PROCESS.md`、`AGENT_LOG.md`、`SPEC.md`、`PLAN.md` 与真实开发过程,力求诚实、不夸大。

## 一、项目定位:为什么是「治理优先」

做这个项目前,我先问自己一个问题:一个编码智能体最稀缺的能力是什么?不是"能写更多代码",而是"在错误的动作真正落地之前把它拦下来"。市面上多数 Agent 演示都把重心放在"能力上限"——让 LLM 尽可能自由地改文件、跑命令;而我选择把重心放在"行为下限"——每一个动作都必须先过治理闸门。于是 AegisCode 被定义为**治理优先**的 harness:命令词法治理、路径围栏、四档判定、人类在环审批、审计哈希链是最深的第一支柱;客观反馈闭环(pytest / 退出码 / 文件变更范围驱动自我修正)是完整但不追求最深的第二支柱。这个取舍贯穿了整个 SPEC 与实现,也决定了后期评审时我最该盯的是哪里。

## 二、方法论:严格的 SDD 工作流买到了什么

整个项目走 Superpowers 的子智能体驱动开发(SDD):每个模块一个 worktree、每个 task 一个全新子智能体、严格 TDD(先写失败测试→确认失败→最小实现→确认通过)、每个 task 后做两阶段评审(SPEC 合规 + 代码质量),且 Critical 问题必须在提交前修掉。这套纪律最大的收益不是"写得快",而是"回归可控"——32 个 task、300 多个测试,每一步都有失败测试先行,后期大改治理引擎时我能靠全量绿灯确认没踩坏别处。它也逼我把"自实现主循环"这件事做实:主循环是一个显式的 `while` 循环(`harness.py`),不依赖任何现成 Agent SDK,因此所有机制在换成 MockLLM 后都能被确定性单测覆盖——这是"harness 自实现"命题唯一诚实的证明方式。

## 三、冷启动与规约演化

`SPEC_PROCESS.md` 记录了 25 轮讨论和五轮冷启动(让陌生智能体试跑规约、暴露缺陷、再修订 SPEC/PLAN)。最有价值的三处,是我推翻智能体推荐的地方:绝对路径策略(第 13 轮)、Agent 能否写记忆(第 16 轮,最终定为 `source=agent`、`confirmed=false`、永不作治理依据)、云部署鉴权场景澄清(第 19 轮)。这三处提醒我:规约不是一次写成的,是被"试错"逼出来的。尤其第四轮冷启动挖出的 fail-open 安全缺陷(敏感文件在符号链接下被绕过),前三轮全没发现——这让我对"看起来没问题"保持了应有的警惕。

## 四、做对了什么

回头看,几个关键决策经受住了最终评审:自实现主循环 + MockLLM 确定性,让每个机制都能离线复验;完成判定交给 `final_verifier` 客观复跑,而非 LLM 自称"我做完了";审计链坚持 redact-before-hash——脱敏发生在计算 SHA256 与落库之前,所以哈希不会算在明文上;凭据状态查询只回掩码,镜像绝不烤 Key。这些不是事后补的,是从 SPEC 阶段就定下并一路守住的。

## 五、踩过的坑:最有价值的教训

最深的教训是**"单测绿 ≠ 端到端接上"**。项目级最终评审集中暴露了这个模式:记忆检索和审批指纹绑定,两者都**建好了、都有单测**,却都**没接进主循环**——`harness` 里 `memories=[]` 是硬编码,审批指纹靠对象同一性"碰巧安全"。如果只看单测覆盖率,它们都是绿的;只有把评审拉到"整条活循环端到端"的粒度,才发现闭环缺了一半。相关的还有 test theater(断言打在头部注释而非真实 CMD 上)和只有红队评审才能发现的命令治理旁路(`python3` 绕过只匹配 `python` 的规则、紧贴式 `-c'...'`、`run_command` 未过路径围栏可 `cat .env`)——这些在"正常路径"测试里全是绿的,是我拿真实默认配置逐条复现才确认的。另一类坑只有**真正在干净环境跑**才会现形:容器里没有 `aegis.yaml` 时环境变量覆盖被静默丢弃,导致 MockLLM 模式起不来;以及 `filterwarnings` eager-import 一个旧版 starlette 没有的类,让干净安装的 pytest 直接崩。这些都不是"写代码时想得到"的,是 docker build / docker run 逼出来的。它们共同指向一个朴素但反复被忽视的事实:测试证明的是"我测到的行为",不是"系统真实的行为";要缩小这道缝,唯一可靠的办法是让评审的粒度贴近真实的端到端路径,并且亲手在最接近生产的环境里跑一遍。这一轮我把评审拆成四个并行的只读子智能体分片覆盖检查清单,再逐条独立复现,正是为了不让任何一个"看起来绿"的机制蒙混过关。

## 六、人机协作的边界

这个项目是在 AI 智能体高度参与下、按严格 SDD 流程完成的,这本身就是过程的一部分,不必回避。但边界很清楚:**治理性的判断留在人这一侧**。修复深度选"彻底加固"还是"最小正确"、超时与脱敏正则是否本轮一并修、记忆是否接入主循环、公网是否部署——这些都是我在评审关口逐项拍板的,子智能体只负责在给定边界内实现与自证。Critical 问题我坚持"提交前必修",且每一个都亲自独立复现,而不是只信子智能体的报告。

## 七、局限与展望

诚实地讲,当前版本是单用户 / 单任务 / 单个本地仓库、锁定 Python + pytest 单栈的最小闭环。记忆是 SQLite 无向量库的最小实现;WebUI 无鉴权、仅限本机,公网部署本轮按决策暂缓;密钥扫描器有意只覆盖已知格式,不是通用探测器。这些都是自觉的取舍而非疏漏。若继续做,我最想补的是把治理决策也纳入记忆(在"永不作治理依据"的前提下做可解释的提示)、以及给 WebUI 加一层真正的鉴权再谈公网。但即便到此为止,这个项目已经把我最初想验证的那件事做实了:一个编码智能体,可以在"自由"之前先学会"被治理"。
