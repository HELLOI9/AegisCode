# REFLECTION.md

## 1. 项目回顾

AegisCode 的目标不是再实现一个以代码生成能力为主要功能的 Coding Agent，而是解决一类更具体的问题：当 LLM 能够读取文件、修改代码和执行命令时，谁来约束它的动作，谁来判断任务是否真正完成，以及这些判断能否脱离 LLM 被重复验证。

围绕这一问题，项目最终形成了两条核心主线。第一条是治理主线：所有动作在执行前必须经过命令词法治理、路径围栏、四档风险判定和人工审批，高风险动作不会因为模型“看起来可信”而被直接放行。第二条是反馈主线：pytest 结果、命令退出码和文件变更范围被标准化后回灌至下一轮，任务完成由 Harness 独立复跑测试决定，而不是由模型自行声明。

项目最终完成了原始计划中的 32 个任务，并在后续迭代中补齐了实际托管平台上的 GitHub Actions、Render 公网部署和 WebUI 三项机制演示入口。项目还追加完成了真实 LLM Provider 可用性，使 OpenAI、Anthropic 和 OpenAI 兼容端点能够真正驱动 Harness。最终基线为 `make test` 通过 444 项测试，`make demo` 的三项机制演示全部通过，公网 Demo Mode 和真实 Provider 端到端验证也完成了独立验收。

结果本身固然重要，但笔者认为，本项目更有价值的部分是设计和实现过程中不断暴露出来的错误：许多问题并非“代码不会运行”，而是“局部测试全部通过，但真实安全保证并未接入完整系统”。这些问题促使笔者重新理解了 AI 辅助软件工程中规格、测试、评审与人工判断的关系。

## 2. 设计阶段的主要收获

### 2.1 设计决策不能完全交给智能体

项目最初希望同时覆盖治理、反馈、可观测性、记忆、多 Provider 和完整 Web 产品形态。按照笔者以往的习惯，可能会直接编写笼统的提示词，让大模型自行实现机制。如果按照这种做法，大模型的决策偶然性可能导致最终得到一个功能较多但每项机制都不够深入的系统，并可能错漏百出。因此本项目笔者使用了Superpowers工作流，在设计决策阶段先进行brainstorm。

Superpowers 的 brainstorming 技能能够持续提出备选方案、成本和风险，对完善设计有明显帮助。经过多轮的对话探讨，笔者明确的项目细节，将项目定位收敛为“治理优先”，把命令治理和路径围栏作为深挖方向，把反馈闭环作为完整但次一级的支柱，其余维度只实现满足课程要求的最小闭环。在后续所有设计中，笔者都严格围绕这一项目定位核心进行，比如严格单动作循环、审批指纹绑定、审计哈希链、MockLLM 确定性测试等等。

值得一提的是，在brainstorming中，笔者并没有一味赞同智能体的决策。项目中有三处典型的人工作出不同选择：

1. 对工作区内的绝对路径，智能体倾向于一律拒绝，笔者选择允许，但要求经过 `realpath + commonpath` 归属判定；
2. 对 Agent 写长期记忆，智能体倾向于完全禁止，笔者选择允许提议写入，同时增加“未确认、不得用于治理、写前脱敏”三重限制；
3. 对公网部署，笔者通过追问区分了“用户在本地使用自己的 Key”和“陌生访客在课程公网服务器上执行代码”两个场景，最终采用本地零鉴权、公网受限 Demo Mode 的不同边界。

这些决策说明，智能体擅长生成方案和补充护栏，但项目目标、可用性与风险之间的最终平衡仍需要开发者承担责任。工程师不能因为某个方案被描述为“最安全”就直接接受，也不能因为某个方案更方便就忽略其执行面风险，要真正起到决策评审的核心作用。

### 2.2 冷启动验证比同一上下文中的自查更可靠

项目在正式实现前进行了五轮冷启动验证：让没有历史对话和隐含上下文的新智能体，仅依据 `SPEC.md` 和 `PLAN.md` 尝试实现关键任务。该过程先后发现了配置结构矛盾、任务编号和依赖不一致、黄金路径中 `pip` 未进入允许列表、测试 helper 无创建归属，以及最严重的零配置 fail-open 问题。

其中最有代表性的技术问题是：危险命令规则一度只存在于示例 YAML 中，代码默认值却为空。只要用户不加载配置，`pip install` 就会从允许列表直接落入 `ALLOW`，完全绕过审批。此前的回归测试又使用了手写规则作为 fallback，使该缺陷继续保持绿色。

这一步骤直观反应了规格文档经过同一个人和同一个智能体多次阅读后，会产生强烈的“已经足够清楚”的错觉问题。项目负责人或者大模型可以通过共享上下文、memory等机制自动补全缺失信息，而陌生实现者不会。因此冷启动环节是必要的，它的价值正在于切断这些隐式补全，使规格更加完善、更为自洽。

此外，在第三次冷启动也暴露了流程问题：验证 worktree 从过时的远端 `origin/main` 创建，导致智能体针对旧版本文档报告了已经修复的问题。

鉴于此，笔者规定智能体的处理流程必须先确认被验证提交已经 push或者明确从本地 HEAD 创建 worktree，再进行程序的修改与测试。如果说Superpowers是一个大的workflow框架，那么这次冷启动暴露的问题就是一个子workflow框架，只有在正确的workflow约束下，智能体才能产生满意的工程结果。

## 3. 实现工作流的反思

### 3.1 Task 粒度与 Subagent 的自主运行边界

在实际开发中，Task 的颗粒度直接决定了 Subagent 能够自主运行多久，以及它偏离目标后需要付出多大修正成本。

笔者最初倾向于将任务拆得非常细，认为步骤越细，模型越不容易出错。但实践中发现，过细的拆分也会产生明显副作用：大量时间被消耗在重复读取上下文、创建 Subagent、提交代码、生成报告和发起评审上，真正用于实现的时间反而下降。特别是仅涉及几行机械修改的任务，如果仍完整执行一次“实现—规格评审—质量评审”的流程，就可能出现形式成本大于工程收益的问题。

相反，如果一个 Task 同时包含多个机制、跨越大量文件并要求一次完成集成，Subagent 又容易在长上下文中遗漏边界条件、跳过日志或混淆完成状态。WebUI 迭代和 HarnessCore 集成阶段出现的上下文溢出，便说明里程碑级目标不能直接作为单个 Subagent 的执行任务。

结合本项目经验，笔者认为较合适的 Task 应满足以下条件：

1. 围绕一个可以独立说明的机制或接口展开；
2. 明确给出需要创建或修改的文件；
3. 输入、输出和与其他模块的依赖关系清晰；
4. 存在可以独立运行的验证命令；
5. 能够形成一个语义完整的 commit；
6. 完成条件不仅包括代码和测试，还包括计划状态与实现日志的同步。

例如，路径围栏、命令词法分析、审批状态机分别作为独立 Task 是合理的，因为每个机制都能通过确定性测试独立验收；而将“命令治理”整体放入一个大 Task，则很可能使词法分析、参数规则、路径治理和执行器之间的细节被混在一起。另一方面，仅仅为了增加一条类型注解而启动完整 Subagent 工作流，也没有必要，可以由控制器在评审后完成小范围机械修正并直接验证。

在上下文充分、接口明确的情况下，一个新的 Subagent 通常能够独立完成“编写失败测试—实现—复跑测试—提交—生成任务报告”的完整 Task，但并不适合在无人检查的情况下连续完成整个 Milestone。随着任务数量增加，它容易遗忘早期约束、错误理解当前分支状态或者把自然语言总结误当成真实进度。因此，笔者最终采用的策略是：Subagent 对单个 Task 负责，控制器对跨 Task 的依赖、进度账本、最终装配和里程碑验收负责。

### 3.2 Subagent 不能被当作可靠的自动流水线

Subagent 提高了并行度，但执行可靠性并不稳定。早期使用较弱模型Claude Haiku时，曾出现代码已经完成却没有提交、没有生成报告或最终消息被截断的情况；WebUI 迭代中也出现多次上下文溢出，部分任务最终由控制器直接完成。真实 Provider 阶段还出现过任务顺序记录错误：某项工作被叙述为“已经评审”，但实际并未执行。

由于笔者严格使用Worktree机制开发，这些问题没有直接破坏代码。笔者最终回到 `git log`、测试结果和工作树状态进行核验，修正智能体对当前工作的偏离。

笔者从中也得到了深刻的教训：自然语言进度描述不能作为项目真相源，真实状态必须以提交、文件和可复跑测试为准。对于长链路 Agent 工作流，控制器必须维护明确的进度账本，并在每个阶段结束时核对，而不能依赖模型对前文的记忆。必须强制模型从进度账本中获取信息，并随时更新同步进度账本。

### 3.3 Worktree、TDD 与两阶段评审的实际价值

项目按里程碑建立独立 Git worktree，并为多数任务采用“实现 subagent + 规格评审 + 代码质量评审”的工作流。每个任务尽量遵循红—绿—重构：先提交失败测试，再实现最小代码，最后进行评审和修复。

这一方式带来了三个明显收益：

* 不同里程碑之间的代码和提交边界清楚，出现问题时容易定位；
* 评审发现的问题能够形成独立修复提交，而不是混入大范围改动；
* `AGENT_LOG.md`、提交记录和验收矩阵共同构成了可追溯证据。

项目中的许多真实缺陷并非由subagent（implementer）主动发现，而是由后续 subagent（reviewer） 找出，例如 JSON 解析器无法处理字符串中的花括号、路径围栏只检查输入路径而未检查符号链接解析后的敏感目标、命令词法层遗漏 glob 和换行、审计链无法检测尾部截断等。

从实际价值上来说，两阶段评审不是形式上的重复，而是与实现阶段不同的认知视角，其为代码的正确产出提供了双保险。

### 3.4 实现日志也需要工程纪律

`AGENT_LOG.md` 记录了任务、技能、TDD 过程、评审意见、人工干预和提交哈希，这为后续验收提供了重要依据。但实现过程中也发生过部分任务日志被压缩成汇总块、文档修改后未及时追加记录、后续再根据 Git 历史重建的情况。

笔者认为这是对模型没有做到严格的约束。“记录过程”本身也是一项需要即时执行的工程任务。事后重建虽然可以恢复事实，但会降低过程证据的可信度，并增加遗漏风险。当控制器的上下文过长而出现注意力涣散时，控制器可能会遗忘笔者最开始对于实现日志的要求，导致压缩实现完每一个Task之后跳过日志的编写。

注意到这个问题后，笔者在`Claude.md`中强制加入“完成代码、提交、测试、更新计划状态、更新日志”这一完成任务完成条件，并强制智能体写入`Memory`中。同时，笔者通过`/compact`保证上下文控制在合理的范围内，并编写更为严格的Prompt约束智能体的行为。

### 3.5 有效的 Prompt 与 Context 策略

笔者总结了一套较为有效的 Prompt 策略，该策略的核心思想是提供与当前 Task 直接相关、能够形成闭环的信息。笔者逐步形成了以下 Context 结构：

1. 当前 Task 的目标及其在整体架构中的位置；
2. 相关 SPEC 条款和 PLAN 中的接口约定；
3. 当前允许修改的文件范围；
4. 已存在模块的真实接口，而不是重新描述一份接口；
5. 必须先运行的失败测试和最终验证命令；
6. 明确的禁止事项，例如不得跳过 TDD、不得修改无关模块；
7. 完成检查表：代码、测试、commit、PLAN 状态和 AGENT_LOG 缺一不可。

这种方式比把完整历史对话直接交给 Agent 更有效。过多历史信息会使模型难以区分当前约束和已经失效的讨论，甚至会让它根据旧方案继续实现。相反，只提供经过确认的 SPEC、PLAN、依赖接口和当前进度账本，可以显著降低上下文噪声。

## 4. 技术实现中的核心教训

### 4.1 安全机制必须存在于确定性执行路径中

AegisCode 的核心原则是：安全判断不能依赖模型是否遵守提示词。命令是否允许、路径是否越界、审批是否有效，都必须由确定性代码判定，并在拒绝时保证工具零执行。

实现阶段多次证明，仅仅“已经写了某个安全模块”并不等于系统已经安全。例如命令治理函数最初没有接入 Dispatcher，路径围栏最初只覆盖文件工具而没有覆盖 `run_command`，审批指纹校验最初只存在于单元测试中而没有进入主循环。这导致了每个模块单独测试都可以通过，但实际动作仍可能绕过它们。

因此，安全机制的验收必须覆盖完整路径：动作解析后是否进入治理、治理结果是否在执行前生效、`DENY` 和 `REQUIRE_APPROVAL` 时工具调用次数是否为零、审批恢复时是否重新验证原动作指纹。项目最终通过真实 HarnessCore 驱动的三项 Demo 和 Dispatcher 级 spy 测试，将这些要求固定为可复跑不变量。

### 4.2 安全边界必须基于真实解析结果，而非表面字符串

路径围栏中最典型的缺陷是：输入路径 `report.txt` 本身不敏感，但它可以通过符号链接指向工作区内的 `.env`。只检查输入字符串和工作区归属仍会放行敏感内容。笔者在最终实现中敲定需要同时对输入路径和解析后的真实目标执行敏感匹配。

命令治理也出现了类似问题。`python3` 可以绕过只匹配 `python` 的规则，`-c'import os'` 可以绕过只匹配独立 `-c` token 的规则，Git 的 `--h` 缩写可以绕过 `--hard` 规则。最终需要对解释器族、紧贴短选项和可接受的长选项缩写进行归一化，同时采用“最严格判定优先”，避免用户调整 YAML 顺序后让较宽松规则覆盖拒绝规则。

这些问题使笔者认识到，安全判断必须基于程序真正会如何解释输入，不能停留在原始字符串的直觉层面。路径要看解析后的对象，命令要看规范化后的参数语义，审批要看规范化动作指纹，必须严格牢记这三条铁律。

### 4.3 单元测试通过不代表真实默认配置正确

项目中多次出现“测试使用手写镜像，实际出厂默认却有问题”的情况。最典型的是 Task 14 的测试手写了包含 `pip` 的允许列表，因此测试能得到 `REQUIRE_APPROVAL`；但真实配置默认曾经漏掉 `pip`，黄金路径实际会被直接 `DENY`。后来回归测试又通过 `or RULES` fallback 掩盖了代码默认规则为空的问题。

很显然，复制的配置只证明测试中的配置正确，不能证明产品真实配置正确。因此，关键回归测试必须直接读取 shipped configuration 和代码默认值，不能复制一份“正确答案”作为 fixture。对于安全默认值，还必须增加“无 YAML、无环境补丁”的零配置测试，验证系统在最少前提下仍然 fail-safe。

笔者也曾经在众多社交媒体上看到软件工程师有关大模型“伪造”测试的技术帖，不少技术帖认为测试环节是目前大模型编码最薄弱的环节，在本项目中笔者也是对这个问题第一次有了清晰的认知。作为软件开发的核心流程，哪怕最顶尖的大模型也会出现假测试的情况，这说明工程师尤其需要把控代码审批的步骤。

大模型的假测试问题应当得到所有工程师的充分重视！

### 4.4 测试必须能够杀死目标缺陷

在真实 Provider 的最初收尾引导中，增加了一条测试：只检查 system prompt 中出现 `finish` 和 `pass`。然而，在原有提示词中实际上已经包含这两个词，即使删除新规则测试仍会通过。这是典型的 test theater：测试看起来覆盖了需求，实际上无法识别目标功能是否存在。

笔者的修复方式是将其改为断言，并新增规则独有的文本，使用 mutation 验证：删除该规则后测试必须失败，恢复后重新通过。

类似地，CI secret scan 也不能只测试底层正则，必须植入真实密钥，走完整目录遍历、allowlist 和退出码路径才算是完整可靠的测试。

笔者认为，高价值测试不应只问“当前代码是否为绿色”，还应问“删掉或破坏被测机制后，它是否会变红”。对于安全和验收关键路径，mutation 思维比简单覆盖率更有意义。

### 4.5 MockLLM 的确定性与真实模型可用性是两个不同问题

课程要求强调移除真实 LLM 后仍能验证核心机制，因此项目从一开始就使用 `MockLLM` 驱动治理、反馈和审批测试。这一方向是正确的：它保证普通测试零网络、零费用、无随机性，并使每个核心机制能够被精确断言。

但后续真实 Provider 追加实现暴露了另一个事实：适配器存在，并不代表真实模型能够驱动 Harness。原实现虽然有 OpenAI 和 Anthropic Adapter，但主循环给模型的 system prompt 和工具协议为空。MockLLM 按预设脚本返回 JSON，不受影响；真实模型却只看到任务文本，因此持续输出散文并触发 `INVALID_ACTION`。

在笔者补充 `PromptBuilder`、动态工具协议和 Provider 无关的系统约束后，真实模型才具备可用性。然而，在首次 DeepSeek 运行又暴露出模型在测试通过后不调用 `finish`，而是重复写入同一文件，最终被 `NO_PROGRESS` 正确终止。该结果说明 Harness 的停机机制有效，但提示词缺少收尾纪律。随后项目增加“不重复成功动作”和“测试通过后立即 finish”的引导，并完善端到端轨迹输出。

这一过程让笔者明确区分了两类验证：

1. MockLLM 应用于证明 Harness 机制的确定性
2. 真实 LLM E2E 用于证明模型能够理解并实际使用这些机制。

前者应进入普通 CI，后者应由人工触发，不能互相替代。

### 4.6 凭据治理必须覆盖完整生命周期

在本项目开始之前，笔者容易把 API Key 理解为一个简单的配置项：从环境变量读取，然后交给 LLM Adapter 即可。课程对凭据工程的要求迫使笔者把问题扩展为完整生命周期：凭据如何录入、存储、读取、展示、清除、注入容器，以及如何避免它进入源码、Git 历史、Shell 历史、日志、异常、WebUI、审计记录和 Docker 镜像。

AegisCode 最终实现了分层凭据方案：

```text
keyring → .env（显式开启）→ 环境变量
```

笔者对凭据的使用做了如下规定：

1. 在普通本地环境中，系统优先使用系统 keyring。
2. `.env` 是明文文件，默认关闭，只有用户明确开启后才能作为降级方案；在容器等没有系统凭据服务的环境中，则允许通过运行时环境变量注入。
3. 凭据通过 `aegiscode key set` 使用 `getpass` 录入，避免回显和进入 Shell 历史；
4. `key status` 只返回配置状态和掩码，`key clear` 负责清除凭据。
5. 选择真实 Provider 却没有可用 Key 时，系统会在发出网络请求前安全失败，而不是带着空值继续运行。

此外，笔者发现仅仅安全存储 Key 还不够。凭据可能通过模型反馈、工具输出、异常堆栈和审计事件再次泄漏，因此项目实现了统一脱敏器，并坚持审计数据必须先脱敏，再计算 SHA256 并写入数据库的原则，防止明文凭据因为“需要保持哈希稳定”而被保留下来。记忆存储在写入前也会调用密钥扫描器，拒绝把 Key 写入长期记忆。

在仓库和 CI 层面，项目同时使用自写的确定性扫描脚本和 gitleaks 兜底。自写扫描器可以用假 Key 离线单测，并被记忆写入等内部机制复用；gitleaks 提供更广泛的外部扫描能力。CI 不只测试正则本身，还需要把测试密钥植入真实扫描路径，确认目录遍历、允许列表和非零退出码均生效。

Docker 分发同样属于凭据边界的一部分：`.dockerignore` 排除 `.env`、私钥、数据库和其他敏感文件；Dockerfile 不通过 `ENV` 或 `COPY` 写入 Key；真实凭据只允许在容器启动时注入。这样即使镜像被分发，也不会造成凭据的泄漏。

凭据工程使笔者认识到，安全不是“不要把 Key 写死”这一条规则，而是一条贯穿输入、内存、日志、持久化、构建和运行环境的数据流。任何一个环节只要返回了明文，就可能抵消其他层的安全设计。对于带真实 LLM Provider 的系统，凭据生命周期必须与功能代码同时设计，而不能在项目收尾时再补一个环境变量读取函数。

## 5. 对计划偏差与后续工作的说明

### 5.1 原计划要求首次未完成，后续补齐

GitHub Actions 和 Render 公网部署都属于原计划要求，不是后续新增需求。

首次实现只完成了 GitLab CI，未在实际托管平台上运行 GitHub Actions；除此以外，Render 公网 URL 也因优先验收核心 Harness 和治理机制而暂缓。

后续工作补齐并完成了 GitHub Actions 三项 job、Render Blueprint、健康检查、Demo Mode 和部署验证命令。

### 5.2 对已有功能的迭代

Render 上线后，WebUI 虽然可以创建任务和处理审批，却无法直接运行 `make demo` 对应的三个 MockLLM 场景。后续增加共享场景层、Demo API 和 WebUI 下拉入口，使 CLI 与 Web 使用同一脚本和同一成功条件。这一工作没有改变 Harness、治理或审批机制，属于对原 Task 28 WebUI 的完善。

### 5.3 真实 Provider 增强

真实 LLM Provider 可用性不是简单增加一个 API 调用，而是补齐系统提示词、动态工具协议、Provider 请求组装、同步审批测试入口、真实 E2E 和收尾可观测性。最终使用 DeepSeek 的 OpenAI 兼容端点完成真实端到端验收，由 Harness 工具创建实现和测试文件，经治理和审批后运行 pytest，并在客观验证通过后进入 `COMPLETED`。

这部分增强验证了架构中的 LLM 抽象是可替换的，也反过来暴露了 MockLLM 无法发现的交互缺陷。

值得一提的是，考虑到安全机制以及使用需求，笔者的公网部署仅仅只作为demo的一种演示，若需实际测试端到端的模型调用需在本地cli中部署测试。

## 6. 项目当前的不足

尽管项目已经完成课程验收，当前实现仍有清晰边界：

* 系统面向单用户、单任务和单个本地仓库，尚未处理并发任务、资源配额和多租户隔离；
* 目标技术栈限定为 Python 与 pytest，尚未抽象通用构建、测试和语言工具链；
* 普通 WebUI 没有身份认证，只适合绑定本机；公网只能运行受限 Demo Mode；
* 路径和命令治理面向 Linux 语义，未处理 Windows 大小写折叠和跨平台命令差异；
* 记忆系统只采用 SQLite、关键词 LIKE 和 topK，不具备复杂语义检索能力；
* 写操作快照回滚仍为 no-op，任务失败后的自动恢复能力有限；
* 密钥扫描器只覆盖已知模式，不是通用秘密检测器；
* SHA256 哈希链可以检测多数篡改，但不提供外部可信锚和签名，尾部截断仍需依赖期望计数；
* 真实模型具有非确定性，不同 Provider 对动作协议和收尾规则的遵守程度可能不同；
* Render 免费实例会休眠并丢失临时数据，当前公网实例只能作为机制展示，而非生产服务。

这些限制多数来自主动范围裁剪，而不是被忽略的问题。笔者认为，在课程周期内明确写出边界，比用不完整实现宣称“通用”更符合工程诚信。

## 7. 后续改进方向

若继续迭代，笔者会优先推进以下工作：

1. **加强运行隔离**：将每个任务放入独立容器或更严格的系统沙箱，限制 CPU、内存、网络和文件系统能力，降低通用命令工具带来的系统风险；
2. **补齐身份与多用户边界**：增加认证、授权、任务所有权和审计访问控制，使 Web 服务能够安全地支持真实用户；
3. **完善回滚与恢复**：实现写前快照、任务级恢复和可验证回滚，而不是保留 no-op 接口；
4. **扩展客观反馈源**：在保持确定性的前提下接入 lint、typecheck、build 和更细粒度的文件变更策略；
5. **提升跨语言能力**：把测试命令、工具注册和项目类型识别抽象为可配置 profile，而不是直接假设 Python + pytest；
6. **强化审计可信度**：研究 HMAC、外部时间戳或远端只追加存储，为哈希链增加独立可信锚；
7. **建立真实 Provider 评估集**：使用固定任务集记录不同模型的协议遵守率、完成率、平均步数、审批触发和无进展次数，但不把这些非确定性指标混入核心单元测试；
8. **改善实时交互**：将前端轮询升级为 SSE，并为长任务增加更清晰的取消、恢复和资源状态展示。

## 8. 对 AI 辅助开发的总体认识

AegisCode 的开发大量使用了 brainstorming、writing plans、worktree、subagent-driven development、TDD 和多轮代码评审。AI 对任务拆分、接口草拟、测试生成和并行评审提供了显著效率，但项目过程也证明，AI 不能替代工程负责人。

Superpowers 的方法论隐含了几个假设：项目能够被拆分为相对独立的 Task；测试能够客观定义完成；Subagent 可以获得足够准确的 Context；Git 和 worktree 能够隔离风险；开发者会持续评审智能体的工作。这些假设在 AegisCode 的大部分模块中成立，因此该方法论总体有效。但在跨模块主循环集成、真实 Provider 交互和公网部署等任务中，接口与环境高度耦合，单 Task 局部正确并不能保证系统整体正确，这时必须增加集成测试、真实运行和人工决策。

智能体会遗漏收尾步骤，会在长上下文中混淆任务状态，会生成看似合理但无法杀死缺陷的测试，也可能因为共享上下文而忽略规格中的歧义。另一方面，智能体也能提出笔者未考虑的风险，并在明确边界下快速给出修复方案。笔者认为，AI 的价值在于扩大分析和执行能力，而人的价值在于决定方向、定义完成标准、识别错误前提，并对最终结果负责。

因此，Superpowers 既是一条开发流水线，更是一套带有自治执行能力的工程纪律框架。它们能够帮助项目执行既定工程流程，但不能代替开发者的责任。项目最终能够完成，不是因为智能体从未出错，而是因为每次错误都可以通过规格修订、回归测试、独立评审和过程记录进行自动纠正并转化为可复用的工程规则！

## 9. 总结

综上所述，AegisCode 最终实现了一个可运行、可测试、可演示的治理优先 Coding Agent Harness，笔者也通过此项目更具体地理解了可靠 Agent 系统的构建方式：

* 模型负责提出动作，Harness 必须掌握执行权；
* 安全规则必须是确定性代码，不能依赖提示词服从；
* 完成状态必须由客观验证产生，不能由模型自报；
* 单元模块存在不等于机制已经接入真实路径；
* 测试必须覆盖真实默认配置，并能够在机制被破坏时失败；
* AI 可以扩大工程能力，但项目的边界、取舍和责任仍属于开发者。

如果重新开始这一项目，笔者仍会保留治理优先、MockLLM 确定性验证和严格单动作循环这三项核心选择，但会更早建立真实默认配置的端到端测试、更早进行项目级红队评审，并把部署与实际托管平台 CI 明确拆成原计划中的独立任务。这样能够减少后期补齐工作，也能更早发现“模块已经完成、系统保证却尚未成立”的集成缝隙。
